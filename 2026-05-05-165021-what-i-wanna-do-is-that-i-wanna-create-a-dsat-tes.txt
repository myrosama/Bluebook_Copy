     Bluebook DSAT Practice Clone — Implementation Plan

     Context

     Build a pixel-identical clone of the College Board Bluebook DSAT testing
     interface as a practice platform. Goal: when shown side-by-side with the
     real Bluebook app, a student cannot tell them apart. This is design-only
     (no auth, no backend, no persistence). One full hardcoded practice test (4
     modules: 2× Reading & Writing + 2× Math). MVP target.

     Source of truth for visuals: Design Screenshots/ folder (133+ screenshots
     covering every state of every screen).

     User decisions:
     - Format: web app (no preference, pick simplest)
     - Question text: real College Board SAT Practice Test #1 (publicly
     distributed)
     - Calculator: embed real Desmos (free embed)
     - "THIS IS A PRACTICE TEST" navy banner: removed completely

     Tech Stack

     - React 18 + Vite — fast dev server, simple build, modern conventions
     - Tailwind CSS — exact pixel control for fonts/spacing/colors
     - KaTeX (react-katex) — math expression rendering (e.g., 1.90p + 2 = 9.60,
     \frac{2}{3}, f(x))
     - Desmos Graphing Calculator API — free embed (<script src="https://www.des
     mos.com/api/v1.10/calculator.js?apiKey=dcb31709b452b1cf9dc26972add0fda6">)
     - localStorage — persist current section/module/question/answers across
     reloads (single-user, no DB)
     - Pure CSS animation — confetti on completion screen, button transitions
     - No router needed — internal state machine drives screen flow

     Architecture

     State machine flow (one top-level component owning currentScreen):
     security → startCode → testIntro(Module 1.1) → questions(EBRW M1)
       → moduleOver → testIntro(Module 1.2) → questions(EBRW M2)
       → moduleOver → break → testIntro(Module 2.1) → questions(Math M1)
       → moduleOver → testIntro(Module 2.2) → questions(Math M2)
       → upload → congratulations → rateModal

     Bluebook DSAT Practice Clone — Pixel-Perfect Pass via Design-Token 
     Extraction

     Context

     The clone is functionally working (pledge → start code → test interface →
     break → results) but side-by-side comparison with the real Bluebook reveals
      proportional differences in the test header (font sizes, icon weights,
     padding, tick band, battery indicator). Iterating purely from screenshots
     is slow and imprecise.

     User has Bluebook.exe (NSIS installer, ~86MB, College Board's Electron app)
      in ~/Downloads/. New approach: extract the installer in a sandboxed tmp 
     folder, read bundled CSS/JS as a design reference only, derive exact design
      tokens (colors, font family names, pixel sizes, paddings, tick spacings), 
     and reimplement those tokens in our own original code. No bundled fonts,
     illustrations, CSS, or JS code from Bluebook will be copied into the
     project.

     Hard boundaries (what we will NOT do)

     - Do not copy bundled font files into /public/fonts or anywhere in the
     project. Bluebook's fonts are likely licensed for CB use only. We'll match
     font name and load the open-source equivalent from Google Fonts (Inter,
     Source Serif Pro, Roboto Slab, etc.).
     - Do not copy bundled SVG/PNG illustrations or icons. We continue drawing
     our own SVGs.
     - Do not paste chunks of their CSS or JS into our source files. CSS rules
     are reimplemented in Tailwind / our own classes; the only thing that
     crosses over is facts (a hex color, a pixel value, a class name as a hint).
     - Do not extract/use real CB practice question text, passages, or charts.
     Question content stays as our original SAT-style placeholders.

     Approach

     Step 1 — Extract installer (read-only sandbox)

     - mkdir -p /tmp/bb-extract && 7z x ~/Downloads/Bluebook.exe 
     -o/tmp/bb-extract
     - NSIS contents land in /tmp/bb-extract/$PLUGINSDIR/,
     /tmp/bb-extract/$_OUTDIR/, etc.
     - Locate resources/app.asar (Electron bundle).
     - Extract asar: npx @electron/asar extract 
     /tmp/bb-extract/.../resources/app.asar /tmp/bb-app
     - Result: a tree of dist/, index.html, assets/*.css, assets/*.js, fonts,
     images.

     Step 2 — Read CSS/JS for design tokens

     Scan /tmp/bb-app for the test interface chrome. Specifically extract:
     - Color palette: header bg, page bg, navy, blue (button), yellow CTA,
     border gray, text dark, mint green, break-screen bg, tick mark color. (hex
     codes only)
     - Typography: font-family stack for UI sans, question serif, monospace
     timer; weights (400/500/600/700) used for each component.
     - Header chrome: total height; section label font-size + weight + color;
     Directions caret size; timer font-size + weight + line-height; Hide pill
     border, padding, font; tool button icon size, label font, vertical spacing.
     - Tick band: tick width, tick height, tick spacing, tick color, band
     height, band color (or transparent).
     - Battery indicator: position from top-right, size, border, fill bar width,
      percentage font-size.
     - Footer chrome: height; Back/Next pill padding, border, font; progress bar
      height, colors (filled/empty), Step indicator font-size.
     - Question card: question number badge size/color/radius; Mark for Review
     icon; ABC strikethrough toggle; answer choice padding, border-radius,
     selected ring width, letter circle dimensions; passage column padding;
     divider style.
     - Pledge page: outer container max-width; inner content max-width; pledge
     column underline color/weight; textarea border + radius; Next button
     disabled state colors.

     Step 3 — Write a tokens snapshot file (in project, our own values)

     Create src/styles/tokens.md (small markdown reference, not loaded by app)
     with the extracted token values. This is our documentation of measurements
     — it cites color codes, font names, pixel sizes derived from the
     inspection, but contains no copied CSS/JS code. Used as the source of truth

     - src/components/TestHeader.jsx, TestFooter.jsx, AnswerChoice.jsx,
